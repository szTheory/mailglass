## Decisions Made

### `inserted_at: nil` sentinel instead of `id: nil`

The plan's verbatim replay-detection pattern came from accrue's bigserial precedent: Postgres generates the id server-side, so on `ON CONFLICT DO NOTHING` no row is inserted, RETURNING yields nothing, and Ecto returns `{:ok, %Event{id: nil}}`. Mailglass uses UUIDv7 autogenerated client-side via the `Mailglass.Schema` macro (D-25/D-28) — so Ecto has ALREADY populated `id` on the changeset before sending the INSERT statement. The struct's `id` is never nil on the conflict path.

The sentinel that IS nil is any DB-defaulted column where the value only comes back via RETURNING. `inserted_at` is declared `default: fragment("now()")` in the Plan 02 migration (and `read_after_writes: true` in the Event schema), so it's nil when ON CONFLICT skips the row. Detection logic, the `append_multi/3` Multi.run tests, and the property test all use `inserted_at` as the sentinel. Moduledoc captures the reasoning in a dedicated "The replay-detection sentinel" section so future maintainers don't re-derive this from scratch.

### Property test sandbox-mode flip

The plan specified `use Mailglass.DataCase, async: false`. DataCase uses `Ecto.Adapters.SQL.Sandbox` in `:manual` mode, which wraps every test in a transaction. TRUNCATE inside that wrapper (over 1000 iterations) either thrashes connection reuse or deadlocks — empirically the test disconnects Postgrex after ~60s. Matched the Plan 02 `migration_test` pattern: `use ExUnit.Case, async: false` with `Sandbox.mode(TestRepo, :auto)` in `setup` and `Sandbox.mode(TestRepo, :manual)` restored in `on_exit`. `:auto` mode removes ownership tracking for this test's lifetime only; other DataCase-using tests in the same suite remain isolated under the transactional sandbox.

### OTel trace_id probe is a stub

The plan's verbatim `current_trace_id/0` probed `Code.ensure_loaded?(:otel_propagator_text_map)` + pattern-matched `{:trace_id, id}` from `current/0`. That pattern doesn't match the actual `:otel_propagator_text_map` return shape. Since Phase 4 is the first phase with a concrete trace-propagation call site (webhook ingest), Plan 05 ships `current_trace_id/0` returning `nil`. Callers can still pass `:trace_id` explicitly; Phase 4 will replace the stub with a tested probe against real OTel context.

## The telemetry contract

**Emitted path:** `[:mailglass, :events, :append, :start | :stop | :exception]`

**:start metadata:** `%{tenant_id: String.t() | nil, idempotency_key_present?: boolean}`
**:stop metadata:** SAME shape as :start (span handler re-emits unchanged map per `Mailglass.Telemetry.span/3` contract)
**:exception metadata:** SAME shape + Telemetry span auto-adds `:kind`, `:reason`, `:stacktrace`

**Forbidden keys (T-02-03 mitigation, enforced by PII-refutation test):** `:recipient, :email, :to, :subject, :body, :html_body, :headers, :from`

**Consumers (Phase 4 hook points):** The webhook handler can read `:stop`'s `idempotency_key_present?` to distinguish fresh inserts from conflict-replays without a second DB query. Phase 4's replay-counter telemetry handler will `:telemetry.attach` to `[:mailglass, :events, :append, :stop]` and increment a counter when `idempotency_key_present?: true` + the event's `inserted_at` is nil (cross-checking against the span's return value).

Note: Plan 05's `append/1` path wraps its entire body (including Repo.transact) in the span, so :stop fires AFTER the transact block returns. `append_multi/3` does NOT wrap its own span; the caller's `Repo.transact/1` block (or Multi.transaction) spans the composed Multi and the caller attaches their own instrumentation. Plan 06 Projector will wrap the combined `Multi.update(:delivery) + append_multi(:event, attrs)` block in `Mailglass.Telemetry.persist_span([:delivery, :update_projections], ...)` per D-04 + Phase 1 D-27.

## Reconciler cadence deferral

Phase 2 ships `find_orphans/1` + `attempt_link/2` as pure query primitives with zero Oban dep. Phase 4 will implement `Mailglass.Oban.Reconciler`:

```elixir
defmodule Mailglass.Oban.Reconciler do
  use Oban.Worker, queue: :mailglass_reconcile, unique: [period: 60]

  def perform(%Oban.Job{args: %{"tenant_id" => tenant_id}}) do
    for orphan <- Mailglass.Events.Reconciler.find_orphans(tenant_id: tenant_id, limit: 100) do
      case Mailglass.Events.Reconciler.attempt_link(orphan) do
        {:ok, {delivery, event}} -> link_and_update_projection(delivery, event)
        {:error, _} -> :skip
      end
    end
    :ok
  end
end
```

## W7 test scoping

**Plan 05 scope:** The `append_multi/3` tests exercise ONLY the happy path — `{:ok, %{event: event}} = TestRepo.transaction(multi)`. Error paths via `Ecto.Multi.transaction/1` return a 4-tuple `{:error, failed_step, reason, changes_so_far}` that does NOT match `Mailglass.Repo.transact/1`'s `{:ok, _} | {:error, _}` spec. Wrapping a Multi inside `Repo.transact/1` therefore requires a normalized-return wrapper that collapses the 4-tuple to `{:error, reason}` before exiting the transact block.

**Plan 06 scope:** Plan 06 will ship the normalized-return wrapper alongside `Mailglass.Outbound.Projector.update_multi/3`. That's the right layer for it because Projector is the first Multi-composing caller that actually mixes `Ecto.Multi.update` (which can fail with constraint errors) + `Events.append_multi` (which cannot fail on INSERT conflict, only on changeset invalidity) inside a single transact boundary. Plan 05's tests exercise `append_multi/3` in isolation via `TestRepo.transaction(multi)` directly.

**Phase 4 scope:** The webhook plug's composed Multi hits the full mixed-failure-mode surface (delivery lookup can 404, event changeset can be invalid, projector update can stale-entry). The Plan 06 normalized-return wrapper plus `Mailglass.Repo.transact/1` is the construction shape; Phase 4 exercises it end-to-end.

## Property-test tuning observations

- **Iteration count:** Fixed at `max_runs: 1000` per ROADMAP §Phase 2 Success Criterion #2. Empirical runtime is ~68 seconds — above the plan's <30s estimate but still within CI budget. The runtime is dominated by N inserts per iteration (sequences can be up to 20 events, replay_count up to 10 → max 200 inserts × 2 passes per iteration); with 1000 iterations that's ~400k inserts total.
- **TRUNCATE vs DELETE vs scoped cleanup:** TRUNCATE is the only option because the BEFORE UPDATE OR DELETE trigger raises SQLSTATE 45A01 on DELETE. A per-tenant scoped wipe via `DELETE FROM mailglass_events WHERE tenant_id = 'prop-test-tenant'` would fire the trigger and fail. TRUNCATE is safe because it's a DDL operation, not a DML operation, and doesn't fire row-level triggers.
- **Shrinking behavior:** StreamData's default shrinker works well on the sequence-length dimension (shrinks toward shorter event lists) and the replay_count dimension (shrinks toward 1). No custom shrinkers needed. On local test run we saw clean execution through all 1000 iterations — no shrink attempts because nothing failed.
- **Snapshot shape:** Compares `Map{idempotency_key => event_type_atom}` rather than full struct equality. The `inserted_at` microsecond drift between fresh/replayed passes would fail naive struct equality; snapshot comparison proves convergence on the stable fields. If Phase 6 needs finer-grained projection convergence (e.g., `last_event_at` on Delivery), those assertions live in Plan 06's projector tests, not here.
