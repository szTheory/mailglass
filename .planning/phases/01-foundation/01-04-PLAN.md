The Message struct does NOT own: `html_body`, `text_body` directly — those live in `swoosh_email.html_body` and `swoosh_email.text_body`. The Renderer populates them by updating the inner `%Swoosh.Email{}`.

**OptionalDeps gateway pattern (CORE-06 — from PATTERNS.md lines 768-853):**
Each gateway module:
1. Has `@compile {:no_warn_undefined, [...]}` as the FIRST line after `defmodule`
2. Declares `@moduledoc` explaining which dep it gates
3. Implements `available?/0` using `Code.ensure_loaded?/1` (or `Code.ensure_loaded?/1` on an Erlang atom for :otel_tracer, :gen_smtp_client)
4. Returns `true` from `available?/0` if the dep is present, `false` otherwise
5. `Mailglass.OptionalDeps.Sigra` is CONDITIONALLY COMPILED: `if Code.ensure_loaded?(Sigra) do ... end`

Note from STACK.md: the optional-dep Hex package for MJML is `:mjml` (NOT `:mrml`). The Elixir module it provides is `Mjml`. The `:gen_smtp` package provides the Erlang module `:gen_smtp_client`.

```elixir
defmodule Mailglass.Message do
  @moduledoc """
  A rendered or partially-rendered email message.

  `Mailglass.Message` wraps `%Swoosh.Email{}` and carries mailglass-specific
  metadata. It is the central data structure passed through the render pipeline
  and returned from `Mailglass.Renderer.render/2`.

  ## Domain Language

  A **Message** is the rendered form of a **Mailable** (a source-level definition).
  The Renderer populates `swoosh_email.html_body` and `swoosh_email.text_body`
  by transforming the HEEx template through the CSS inliner and plaintext extractor.

  ## Fields

  - `:swoosh_email` — the inner `%Swoosh.Email{}` struct. All email content
    (to, from, subject, html_body, text_body, headers, attachments) lives here.
    Mailglass never bypasses Swoosh for content — it enriches the inner struct.
  - `:mailable` — the adopter module that built this message (e.g. `MyApp.UserMailer`).
    Used for telemetry, the `Mailglass-Mailable` header, and preview auto-discovery.
  - `:tenant_id` — multi-tenant scope. Carried on every record (CORE-03, D-09).
    `nil` in single-tenant mode.
  - `:stream` — message stream: `:transactional`, `:operational`, or `:bulk`.
    Auth and security emails must use `:transactional` (no tracking, per D-08).
    Default: `:transactional`.
  - `:tags` — free-form string tags for filtering and analytics. Default: `[]`.
  - `:metadata` — adopter-supplied extras (PII-free; not included in telemetry).
    Default: `%{}`.

  ## Examples

      iex> %Mailglass.Message{
      ...>   swoosh_email: %Swoosh.Email{subject: "Welcome!"},
      ...>   mailable: MyApp.UserMailer,
      ...>   tenant_id: "tenant_123",
      ...>   stream: :transactional
      ...> }

  """

  @type t :: %__MODULE__{
          swoosh_email: Swoosh.Email.t(),
          mailable: module() | nil,
          tenant_id: String.t() | nil,
          stream: :transactional | :operational | :bulk,
          tags: [String.t()],
          metadata: %{atom() => term()}
        }

  defstruct [
    :swoosh_email,
    :mailable,
    :tenant_id,
    stream: :transactional,
    tags: [],
    metadata: %{}
  ]

  @doc """
  Creates a new `Mailglass.Message` wrapping the given `%Swoosh.Email{}`.

  ## Options

  - `:mailable` — the module that built this message
  - `:tenant_id` — tenant scope
  - `:stream` — `:transactional` (default), `:operational`, or `:bulk`
  - `:tags` — list of string tags
  - `:metadata` — map of adopter-supplied extras

  ## Examples

      iex> email = %Swoosh.Email{subject: "Welcome"}
      iex> msg = Mailglass.Message.new(email, mailable: MyApp.UserMailer)
      iex> msg.stream
      :transactional
  """
  @doc since: "0.1.0"
  @spec new(Swoosh.Email.t(), keyword()) :: t()
  def new(%Swoosh.Email{} = swoosh_email, opts \\ []) do
    %__MODULE__{
      swoosh_email: swoosh_email,
      mailable: opts[:mailable],
      tenant_id: opts[:tenant_id],
      stream: opts[:stream] || :transactional,
      tags: opts[:tags] || [],
      metadata: opts[:metadata] || %{}
    }
  end
end
```

```elixir
defmodule Mailglass.OptionalDeps do
  @moduledoc """
  Namespace for optional dependency gateway modules.

  Each submodule gates one optional dependency behind a `Code.ensure_loaded?/1`
  check and exposes `available?/0`. The `@compile {:no_warn_undefined, ...}`
  declaration in each module (and in `mix.exs` `elixirc_options/0`) suppresses
  compiler warnings when the dep is absent.

  ## Gateway Modules

  - `Mailglass.OptionalDeps.Oban` — gates `{:oban, "~> 2.21"}`
  - `Mailglass.OptionalDeps.OpenTelemetry` — gates `{:opentelemetry, "~> 1.7"}`
  - `Mailglass.OptionalDeps.Mjml` — gates `{:mjml, "~> 5.3"}` (Rust NIF)
  - `Mailglass.OptionalDeps.GenSmtp` — gates `{:gen_smtp, "~> 1.3"}`
  - `Mailglass.OptionalDeps.Sigra` — gates `{:sigra, "~> 0.2"}` (conditionally compiled)
  """
end
```

```elixir
defmodule Mailglass.OptionalDeps.Oban do
  @moduledoc """
  Gateway for the optional Oban dependency (`{:oban, "~> 2.21"}`).

  When Oban is present, `available?/0` returns `true` and callers may safely
  reference Oban modules. When absent, `Mailglass.Outbound.deliver_later/2`
  falls back to `Task.Supervisor` with a `Logger.warning` emitted at boot.
  """

  @compile {:no_warn_undefined, [Oban, Oban.Worker, Oban.Job]}

  @doc "Returns `true` when `:oban` is loaded in the current runtime."
  @doc since: "0.1.0"
  @spec available?() :: boolean()
  def available?, do: Code.ensure_loaded?(Oban)
end
```
