# Plan 06-03: Domain-Critical Checks (LINT-03, TRACK-02)

## Goal

Implement the two highest-stakes domain checks — tenant scope enforcement and auth-stream tracking prohibition. These are the checks most likely to prevent real security incidents (cross-tenant data leak, PII exposure via tracking on auth messages). They are also the most complex, requiring function-name heuristics and call-chain awareness.

## Check Details

### LINT-03: NoUnscopedTenantQueryInLib

The most complex check. Flags any `Repo.all/1`, `Repo.one/1`, `Repo.get/2`, `Repo.get!/2`, `Repo.get_by/2`, `Repo.get_by!/2`, `Repo.delete/1`, `Repo.update/1`, `Repo.insert/1` call in `lib/mailglass/` where:
1. The query target is a tenanted schema (`Mailglass.Outbound.Delivery`, `Mailglass.Events.Event`, `Mailglass.Suppression.Entry`, `Mailglass.Webhook.WebhookEvent`)
2. The query does NOT appear within a function body that also calls `Mailglass.Tenancy.scope/2`
3. The query does NOT satisfy the explicit bypass contract (`scope: :unscoped` + companion tenant-bypass audit telemetry emit or approved helper call)

**Implementation approach:**
- Walk each function definition (`def`/`defp`) in the source file
- Within each function body, collect all Repo call sites targeting tenanted schemas
- Check if `Mailglass.Tenancy.scope` appears anywhere in the same function body
- If a call uses `scope: :unscoped`, require a same-function companion audit telemetry emit (`:telemetry.execute/3` tenant-bypass event) or approved helper call (e.g., `Mailglass.Tenancy.audit_unscoped_*`)
- If neither scoped path nor valid unscoped+audit path exists, flag the Repo call as an issue
- Support `# credo:disable-for-next-line Mailglass.Credo.NoUnscopedTenantQueryInLib` for intentional bypasses (with Credo's built-in mechanism)

**Unscoped bypass contract (TENANT-03):**
- Allowed only when both are present in the same function:
  1. explicit `scope: :unscoped` on the tenant read call, and
  2. explicit tenant-bypass audit telemetry emit (direct `:telemetry.execute/3`) or adjacent approved helper call that emits it.
- Fixture coverage required in `no_unscoped_tenant_query_in_lib_test.exs`:
  - **negative fixture**: `scope: :unscoped` without audit emit -> flagged
  - **positive fixture**: `scope: :unscoped` with audit emit/helper -> not flagged

**Tenanted schema list** (configurable via check params):
```elixir
[
  Mailglass.Outbound.Delivery,
  Mailglass.Events.Event,
  Mailglass.Suppression.Entry,
  Mailglass.Webhook.WebhookEvent
]
```

### TRACK-02: NoTrackingOnAuthStream

Flags any mailable function whose name matches auth-context heuristics AND whose body contains a `tracking:` keyword with any truthy value.

**Auth-context heuristics** (function name contains any of):
- `magic_link`
- `password_reset`
- `verify_email`
- `confirm_account`
- `reset_token`
- `verification_token`
- `confirm_email`
- `two_factor`
- `2fa`

**Detection logic:**
1. Walk each `def` in modules that `use Mailglass.Mailable`
2. Check if the function name atom, converted to string, contains any auth heuristic
3. If yes, walk the function body for `tracking:` keyword in any map/keyword list
4. If `tracking:` is found with a non-false/non-nil value, flag

**Edge cases:**
- `tracking: false` and `tracking: []` are NOT flagged (explicitly disabling is fine)
- Functions in modules without `use Mailglass.Mailable` are not checked
- The heuristic list is configurable via check params for adopter customization
