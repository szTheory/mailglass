# Plan 06-04: Boundary Enforcement Extension

## Goal

Extend the existing `use Boundary` declarations from the Phase 1 foundation (root `Mailglass` + `Mailglass.Renderer`) to cover the three remaining major subsystems: `Outbound`, `Events`, and `Webhook`. This completes the CORE-07 requirement and provides compile-time structural enforcement of the module dependency DAG.

## Current State

Already declared:
- `Mailglass` (root) — `deps: []`, exports a broad surface
- `Mailglass.Renderer` — `deps: [Mailglass]`, pure (no Repo, no Outbound, no processes)

## New Declarations

### Mailglass.Outbound
```elixir
use Boundary, deps: [Mailglass], exports: [Delivery, Projector, Worker]
```
- **Allowed to reach**: `Mailglass.*` exported modules (Config, Telemetry, Message, etc.)
- **Must NOT reach**: `Mailglass.Events` directly (writes go through Repo + Multi, not function calls to Events module)
- **Must NOT reach**: `Mailglass.Webhook` (send path doesn't know about ingest path)

### Mailglass.Events
```elixir
use Boundary, deps: [Mailglass], exports: [Event, Reconciler]
```
- **Allowed to reach**: `Mailglass.*` exported modules
- **Must NOT reach**: `Mailglass.Outbound` (no cycles — events don't trigger sends)

### Mailglass.Webhook
```elixir
use Boundary, deps: [Mailglass, Mailglass.Events], exports: [Plug, Router, CachingBodyReader]
```
- **Allowed to reach**: `Mailglass.*` + `Mailglass.Events.*` exports (webhook ingest writes events)
- **Must NOT reach**: `Mailglass.Outbound` (ingest doesn't trigger sends)
- **Leaf**: No other mailglass boundary depends on Webhook

### Root Mailglass Exports Update

The root boundary's `exports:` list must be updated to include all modules that sub-boundaries need to call into. Review each sub-boundary's actual `alias` and function calls to build the minimal export set.

## Boundary Contract Test

```elixir
defmodule Mailglass.BoundaryTest do
  use ExUnit.Case, async: true

  # Verify the boundary graph is what we expect by compiling
  # and checking that no boundary warnings were emitted.
  # The real enforcement is `mix compile --warnings-as-errors`,
  # but this test documents the intended DAG for future maintainers.

  test "Renderer has no dependency on Outbound or Repo"
  test "Events has no dependency on Outbound"
  test "Adapters do not reach back into Outbound"
  test "Webhook depends on Events but not Outbound"
end
```

## Potential Violations

Existing code may have cross-boundary calls that were fine without boundary declarations. Common patterns to check:
- `Mailglass.Webhook.Ingest` calling `Mailglass.Events.Event` functions → allowed (Webhook deps on Events)
- `Mailglass.Outbound.Projector` writing Event rows via Ecto.Multi → allowed if it uses the schema struct directly (data flow, not module call)
- Mix tasks classified to root boundary via `classify_to:` (already done for `mix mailglass.reconcile` and `mix mailglass.webhooks.prune`)
