**User's choice:** A. Fix-then-publish in 07.1 (Recommended)

**Notes / Rationale (synthesized from advisor agent #1):**

The audit explicitly recommends Option A at `v0.1-MILESTONE-AUDIT.md` lines 152–158. Ecosystem precedent is unanimous: Phoenix LiveView, Oban, Swoosh, Broadway, LiveDashboard all shipped v0.1.0 with working installers. The single counterexample, Igniter (an installer-focused lib), shipped 0.1.0 → 0.1.5 in a single day patching installer bugs.

mailglass's positioning makes broken-installer-on-publish uniquely toxic for three converging reasons:
1. The installer IS the headline marketing claim ("zero to first preview <5 min")
2. v0.1.0 is explicitly framed as a *validation release* — adopters churning on `UndefinedFunctionError` poison the validation signal because they never reach the product
3. Brand voice ("clear, exact, confident not cocky") is structurally incompatible with shipping a "Known Limitations: the installer doesn't compile" CHANGELOG entry

Hex publish is also unrescuable after 24 hours — a broken 0.1.0 burns the version number forever.

Path C (rc cycle) has strong precedent (Beacon CMS, Surface, LiveView 1.0) but is overkill: rc cycles validate large unstable surfaces, while mailglass's blockers are 4 well-scoped installer bugs the audit already diagnoses line-by-line. Path D doubles planning overhead for the same end state. Path B torches first-impression in exchange for shipping a few days sooner — the wrong trade for a single-maintainer validation release.

- Deployment branch policy must include both `mailglass-sibling-group-v*` tag pattern AND `main` branch (latter required for workflow_run firing)
