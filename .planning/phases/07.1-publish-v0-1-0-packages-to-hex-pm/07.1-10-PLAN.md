    **Determining the SHA value:** the SHA must be the FIRST commit on the mailglass main branch — i.e. the commit BEFORE which there is no project history release-please should consider. Run:

    ```
    git log --reverse --format=%H | head -1
    ```

    This returns the very first commit SHA (the project's initial commit). That is the correct value: every commit after this initial commit, up to the merged state of plans 01-09, becomes part of the v0.1.0 CHANGELOG window.

    **Why the FIRST commit and not (e.g.) the v0.1 milestone start commit:** release-please's `bootstrap-sha` semantics is "release-please will consider commits AFTER this SHA as candidates for the next version" (per the customizing.md doc). Setting it to the first commit ensures the CHANGELOG narrative reflects the FULL project history through v0.1.0 — which is correct for the v0.1.0 release because there have been no prior releases to bound the window. After v0.1.0 ships, this field is ignored (the manifest version becomes the windowing authority).

    **Resulting file structure:**

    ```json
    {
      "bootstrap-sha": "<INITIAL_COMMIT_SHA>",
      "packages": {
        ".": {
          "package-name": "mailglass",
          "release-type": "elixir",
          "component": "mailglass",
          "changelog-path": "CHANGELOG.md"
        },
        "mailglass_admin": {
          "package-name": "mailglass_admin",
          "release-type": "elixir",
          "component": "mailglass_admin",
          "changelog-path": "CHANGELOG.md"
        }
      },
      "plugins": [
        {
          "type": "linked-versions",
          "groupName": "mailglass-sibling-group",
          "components": ["mailglass", "mailglass_admin"]
        }
      ],
      "$schema": "https://raw.githubusercontent.com/googleapis/release-please/main/schemas/config.json"
    }
    ```

    Note: place `bootstrap-sha` FIRST (before `packages`) for legibility — it's the most-changed field in this PR and its placement at the top makes the diff self-explanatory in code review.

    **Procedure:**
    1. Capture the initial SHA: `INIT_SHA=$(git log --reverse --format=%H | head -1)`
    2. Edit the file: insert `  "bootstrap-sha": "${INIT_SHA}",` as the first key inside the top-level object.
    3. Validate JSON: `python3 -m json.tool release-please-config.json`.
    4. Sanity-check the SHA is reachable: `git cat-file -e ${INIT_SHA}` (exits 0 if the commit exists).

    **Anti-patterns to avoid:**
    - DO NOT use `release-as` here. Per D-10 the `Release-As:` commit trailer is preferred — it self-cleans, while `release-as` in config pins forever and creates a footgun on the next release cycle.

    - DO NOT use the SHA of a recent commit "to skip old history" — that omits valid CHANGELOG content. release-please's auto-CHANGELOG only summarizes Conventional Commits; non-conventional commits are skipped naturally.
    - DO NOT add the field as nested inside `packages.".".` — `bootstrap-sha` is a top-level config option, not per-package.
