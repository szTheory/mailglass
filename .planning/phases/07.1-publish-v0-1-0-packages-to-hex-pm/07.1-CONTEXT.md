# Phase 07.1: Publish v0.1.0 packages to Hex.pm - Context

**Gathered:** 2026-04-25
**Status:** Ready for planning
**Source:** Research-driven discuss-phase (4 parallel advisor agents synthesized into a coherent plan)

<domain>
## Phase Boundary

Phase 07.1 closes the v0.1 milestone audit gaps and ships the first Hex.pm release of `mailglass` v0.1.0 + `mailglass_admin` v0.1.0 (linked sibling versions per D-01). This is a single coherent phase: installer blocker fixes + verification scaffolding + publish ceremony, in that exact order, so the published artifact reflects what adopters actually install.

The phase was rescoped from "publish only" to "fix-then-publish" after research synthesis: ecosystem precedent (Phoenix LiveView, Oban, Swoosh, Broadway, LiveDashboard) is unanimous that v0.1.0 ships with a working installer; the one counterexample, Igniter, shipped 0.1.0 → 0.1.5 in a single day patching installer bugs. mailglass's positioning makes broken-installer-on-publish uniquely toxic because the installer IS the headline marketing claim ("zero to first preview <5 min"). v0.1.0 is explicitly framed as a *validation release* — the signal "does this fill a real need?" gets corrupted if adopters churn on `UndefinedFunctionError` before reaching the product. Hex publish is also unrescuable after 24 hours: a broken 0.1.0 burns the version number forever.

**Out of scope (this phase):** marketing announcements / Discord / Forum / social posts (handle elsewhere, post-publish), v0.1.x patch planning, v0.2 feature work, mailglass_inbound v0.5 work, GPG-signed git tags, SBOM generation, multi-owner Hex transition.

### Publish Trigger Sequence (release-please bootstrap)

- **D-09:** Add `bootstrap-sha` to `release-please-config.json` pointing at the SHA where the v0.1.0 CHANGELOG window should start. This is a one-shot value (read on first run, then ignored forever) — get it right before merging the first release PR.

  The `Release-As:` trailer self-cleans after merge. **Preferred over** putting `release-as` in `release-please-config.json`, which pins forever.

- **D-11:** Tag naming: linked-versions plugin with `groupName: "mailglass-sibling-group"` produces a single group tag `mailglass-sibling-group-v0.1.0` (one tag for the group, not per package). After the first cut, set `source_ref_pattern` in both `mix.exs` `package/0` blocks so HexDocs "View Source" links resolve correctly.

### publish-hex.yml Hardening

- **D-12:** **Tighten the trigger gate.** Current `if: ${{ github.event.workflow_run.conclusion == 'success' }}` fires on every successful release-please run including PR-update runs. Replace with: gate on `(workflow_dispatch) || (workflow_run AND conclusion == 'success' AND head_branch matches release tag pattern)`. Without this gate, the workflow could double-publish during the first release window.

- **D-13:** **Add `workflow_dispatch` trigger** as permanent safety valve. Inputs:
  - `tag` (string, required) — git tag/ref to publish from
  - `dry_run` (boolean, default `false`) — run `mix hex.publish --dry-run` instead of `--yes`
  - `package` (string, optional, choices `mailglass | mailglass_admin | both`, default `both`) — re-publish single package after partial failure
  Required regardless of whether automatic flow works the first time — `workflow_run` chains are notoriously flaky on first runs.

- **D-14:** **Serialize the matrix** — replace `strategy.matrix` parallelism with two sequential jobs:
  - `publish-core` publishes `mailglass`
  - `publish-admin needs: publish-core` publishes `mailglass_admin`
  Insert a Hex-index polling step between them: `until mix hex.info mailglass 0.1.0 2>/dev/null; do sleep 15; done` (max 5 min, then fail). mailglass_admin's `{:mailglass, "== 0.1.0"}` constraint can only resolve once Hex.pm has indexed mailglass core; parallel publish is a guaranteed race.

- **D-15:** **Move `mix mailglass.publish.check` summary emission to a SEPARATE prior job** (no environment gate). The job writes to `$GITHUB_STEP_SUMMARY`. Reviewer sees the summary in the workflow run page **before** clicking Approve in the `hex-publish` environment, not after. The publish job consumes the summary job via `needs:`. Approve click becomes informed instead of rubber-stamped.

- **D-16:** Require green `ci.yml` on the tagged commit before publish-hex runs. Currently publish-hex only requires release-please success — but main CI runs on the same SHA and is the only thing asserting `priv/static/` is freshly built, no boundary violations, no warnings, etc. Use `actions/github-script` to fetch the latest `ci.yml` run for the tagged SHA and gate on its conclusion.

### `mix mailglass.publish.check` Expansion (15-step procedure)

- **D-17:** Expand `lib/mix/tasks/mailglass.publish.check.ex` from current ~58 LOC denylist-only to the procedure below. Accept `--package mailglass | mailglass_admin` flag; if invoked at repo root with no flag, run for both sequentially.

  1. Run `mix hex.build --unpack --output _publish_check/<pkg>` in the target package directory. For `mailglass_admin`, set `MIX_PUBLISH=true` first so `mailglass_dep/0` flips to the Hex constraint.
  2. Capture file list, sizes, total bytes. Sort, write to `_publish_check/<pkg>-files.actual`.
  3. **Files allowlist diff** against checked-in golden at `.planning/publish/<pkg>-files.expected`. Fail loudly with a diff on any deviation (add/remove). This is the inverse of the audit's lesson — assert against expected, not denylist.
  4. **Forbidden-files denylist** as belt-and-suspenders. Patterns: `_build/`, `deps/`, `.git/`, `.gsd/`, `.planning/`, `.claude/`, `.elixir_ls/`, `cover/`, `node_modules/`, `priv/native/`, `.env*`, `.DS_Store`, `*.beam`, `*.dump`. For `mailglass_admin`: also exclude `assets/` (source CSS — only `priv/static/` ships per D-04 from Phase 7).
  5. **Tarball size guardrails.** Total uncompressed: warn at 4 MB, fail at 6 MB (Hex hard limit 8 MB).
  6. **Required-files presence.** `LICENSE`, `README.md`, `CHANGELOG.md`, `mix.exs` in all tarballs. For `mailglass_admin` also: `priv/static/app.css` ≥ 1024 bytes (catches empty-CSS regression where Tailwind didn't run); all woff2 fonts present; `priv/static/mailglass-logo.svg` present.
  7. **CHANGELOG section presence.** Read `CHANGELOG.md`; regex `~r/^##\s*\[?v?#{Regex.escape(version)}\]?/m`; assert match found and section body non-empty.
  8. **Mix metadata sanity.** `description` present, ≤300 chars, doesn't start with whitespace, doesn't end with period. `package[:licenses]` exactly `["MIT"]`. `package[:links]` includes `"GitHub"`; for `mailglass_admin` also `"HexDocs"`. `source_url` matches `homepage_url` and points to the GitHub repo. `@version` is valid SemVer and matches the manifest entry for that package.
  9. **Parse unpacked `mix.exs` AST** (`Code.string_to_quoted!`), walk for forbidden dep shapes — fail on any `path:`, `git:`, or `github:` keyword in deps list.
  10. **For `mailglass_admin`, validate the linked-version constraint** resolves correctly. With `MIX_PUBLISH=true` set, parse `mailglass_dep/0` output; assert it equals `{:mailglass, "== <X>"}` where `<X>` exactly matches mailglass core's published `@version`. Catches release-please linked-versions plugin misfires.
  11. **No `:dev`/`:test` deps leak.** Run `mix deps.get --only prod` in a temp dir against the unpacked tarball mix.exs; assert clean.
  12. **Compile-from-tarball.** `cd _publish_check/<pkg> && mix compile --no-optional-deps --warnings-as-errors` in a fresh `MIX_HOME`. Catches `elixirc_paths(:test)` leaks where in-tree compile masks missing `lib/` files.
  13. **`mix hex.audit` clean.** Non-zero exit fails the gate.
  14. **`mix hex.outdated --within-requirements`** advisory only — print to summary, don't block.
  15. **Reviewer summary** as `$GITHUB_STEP_SUMMARY` markdown: package name, version delta (`vPREV → v0.1.0`), file count, total size, CHANGELOG section excerpt, top 10 files by size, full file list collapsed in `<details>`.
  16. **Cleanup `_publish_check/`** unless `--keep` passed.

- **D-18:** Output style follows mailglass brand voice — "thoughtful maintainer, clear, exact, confident not cocky." Specific actionable failure messages:
  > `Delivery blocked: package files diff. Files added since last allowlist update: [.elixir_ls/cache/anything.term]. Either add to .planning/publish/mailglass-files.expected or exclude in mix.exs :package :files.`

  Avoid emoji, no panic, no apology. Status verbs (`create`, `update`, `unchanged`, `conflict`) consistent with existing mailglass installer task style.

### Post-Publish Smoke

- **D-19:** **NEW file** `.github/workflows/post-publish-smoke.yml`. Triggers:
  - `workflow_run: ["publish-hex"] types: [completed]` — runs after each successful publish
  - `schedule: cron: "0 12 * * *"` — daily for 7 days post-tag (jobs guard with `if: now - latest_release_date < 7 days`)
  - `workflow_dispatch` — manual re-run with optional `version` input

- **D-20:** Smoke job structure:
  1. **`wait-for-index`** — poll `mix hex.info mailglass <version>` with 30s sleep, max 10 retries (5 min total). Same for `mailglass_admin`.
  2. **`wait-for-hexdocs`** — `curl -fsI https://hexdocs.pm/mailglass/<version>/` with 60s sleep, max 10 retries (10 min total). Same for admin.
  3. **`consumer-install`** — in tmp dir:
     - `mix archive.install hex phx_new --force`
     - `mix phx.new sandbox --module Sandbox --app sandbox --no-ecto --no-mailer --install`
     - Add `{:mailglass, "~> 0.1"}, {:mailglass_admin, "~> 0.1"}` to deps
     - `mix deps.get && mix mailglass.install --yes && mix compile --warnings-as-errors`
     - Assert exit 0; assert no `UndefinedFunctionError` in stderr (literal G-1 regression test, on the published artifact)
     - Boot endpoint: `mix phx.server &`, `curl -fs http://localhost:4000/dev/mail/`, kill server (validates installer's router mount actually serves)
  4. **`retracted-check`** — `mix hex.info mailglass 0.1.0 | grep -v "Retired"`; fail if retracted between publish and smoke.
  5. **`report`** — on failure, open a GitHub issue tagged `publish-smoke-failed` with logs attached.

- **D-21:** Phoenix host required (not bare `mix new`) because `mix mailglass.install` patches Phoenix endpoint and router (per G-3 mitigation in D-02). Bare project lacks the targets.

- **D-22:** Why a separate workflow (not steps inside `publish-hex.yml`): publish-hex.yml is gated by `environment: hex-publish` reviewers. Multi-minute smoke steps inside that workflow either block reviewer-approval window or run silently after Approve. Separating concerns: smoke runs as a regular workflow with normal logging; daily cron reuses it without re-entering the protected environment.

### Governance + Recovery

- **D-26:** **Solo Hex owner** (`szTheory --level full`) for both packages at v0.1. Multi-owner transition deferred to v0.5 milestone when production adopters exist (rationale: at v0.1 the asymmetry — co-owner can `mix hex.publish` from their machine bypassing GH governance — is a worse footgun than the bus-factor risk it solves).

- **D-27:** GitHub Environment `hex-publish` settings — UI-only, configured at github.com/szTheory/mailglass/settings/environments/hex-publish:

  | Setting | Value | Reason |
  |---|---|---|
  | Required reviewers | `szTheory` | Single-maintainer reality |
  | Prevent self-review | **OFF** | GitHub silently disables when only 1 reviewer; document the gap honestly in MAINTAINING.md rather than pretending the gate is two-eyes |
  | Wait timer | 0 minutes | Pre-publish check already runs; wait timer is theatre |
  | Deployment branch policy | "Selected branches and tags" → `mailglass-sibling-group-v*` tag pattern AND `main` branch | Tag for actual publish; `main` required because `workflow_run` fires on the workflow's branch context |
  | Environment secrets | `HEX_API_KEY` (write-API rights) | **NOT** as repo secret; verify under `/settings/secrets/actions` that no repo-level entry exists. Repo secrets exposed to PR jobs are the #1 secret-leak vector. |
  | Environment variables | None | Workflow has all non-secret config inline |
  | Custom protection rules | None | Over-engineered for v0.1 |

- **D-28:** **Retract policy** as 5-line decision tree, embedded in MAINTAINING.md:
  1. Data-loss / security / signature bypass / fails to compile → `mix hex.retire <pkg> <ver> security|invalid --message "<140 chars>"` AND ship `<ver+1>` immediately.
  2. User-visible breakage with workaround → DO NOT retire. Patch within 7 days. CHANGELOG entry.
  3. Cosmetic / docs / non-runtime → DO NOT retire. Roll into next planned patch.
  4. Published <60 min ago, zero downloads → `mix hex.publish --revert <ver>` (only window where unpublish works — also bounded by 24h initial-release window).
  5. Already retired and false alarm → `mix hex.retire <pkg> <ver> --unretire`.

  Bias toward patch over retract. Three retractions in your first six months tells evaluators "don't bet on this lib."

- **D-29:** **Replace placeholder email** in `SECURITY.md` (`security@example.com`) with real disclosure address before publish. Update SLA to honest single-maintainer numbers: "ack within 72 hours, mitigation or workaround within 14 days for critical, advisory published with fix." Current 24h/7d numbers are aspirational. If no security email available, fall back to GitHub Private Vulnerability Reporting (Settings → Security → enable PVR).

  2. **Retract Decision Tree** — D-28 verbatim.

     1. Verify CI green on `main` for the SHA to be released.
     2. Merge release-please PR.
     3. Approve `hex-publish` deployment in GitHub Environment UI.
     4. Within 60 minutes: `mix deps.get && mix mailglass.install` in a fresh Phoenix app to smoke-test (last revert window).
     5. Post release link to Elixir Forum #libraries section.

## Specific Ideas

- **Fix-test sequence is THE single most important DX detail (D-02 / D-04):** drop the simulation FIRST so failing tests honestly reveal G-1/G-3 before patching them. Anti-pattern to avoid: fix templates first while the simulated fixture still passes — leaves G-2/G-5 latent for the next regression and reproduces the audit's exact failure mode.
- **`Release-As:` commit footer self-cleans (D-10);** putting `release-as` in `release-please-config.json` instead pins forever and creates a future footgun on the next release cycle.
- **Files allowlist diff against checked-in golden (D-17 step 3) is the inverse of the audit's lesson** — assert against an EXPECTED list, not a denylist regex. The audit failure mode (G-2/G-5: simulation passing while real artifact broken) recurs in any "denylist of forbidden paths" model because new paths slip through silently. Both checks (allowlist and denylist) belong in the procedure for defense-in-depth.
- **`prevent_self_review` is silently disabled by GitHub when there's only one reviewer (D-27).** Don't pretend the gate is two-eyes; document honestly in MAINTAINING.md "Bus Factor & Continuity" section.
- **`HEX_API_KEY` environment-vs-repo-scope is the #1 secret-leak vector (D-27).** Verify under `/settings/secrets/actions` — no repo-level entry should exist; only the `hex-publish` environment scope.
- **60-minute post-publish smoke install in a fresh Phoenix app is the maintainer's last revert window (D-30 step 4).** Set a literal timer when approving the deployment.
- **mailglass_admin race (D-14):** `{:mailglass, "== 0.1.0"}` cannot resolve until Hex.pm has indexed mailglass core. Serialize the matrix — parallel publish is a guaranteed race even if it succeeds 90% of the time. Use `needs:` + Hex-index polling, not `max-parallel: 1`.
- **HexDocs cross-refs (D-23):** `.planning/ROADMAP.md` and `.planning/PROJECT.md` are project-internal planning artifacts — they belong on the GitHub repo, not in adopter-facing HexDocs. Removing them from `extras` keeps the published artifact lean.
- **Reviewer summary BEFORE Approve, not after (D-15):** GitHub Environment review approval shows only the workflow YAML and triggering commit. Reviewer has no idea what's in the tarball. Step summaries on environment-gated jobs render *after* approval. Workaround: run `mix mailglass.publish.check` in a SEPARATE prior job (no environment gate, just a normal job) that writes the summary; the gated `publish` job depends on it. Reviewer sees the summary in the workflow run page before clicking Approve.
- **HEX_API_KEY is provisioned** with write API rights on github.com/szTheory/mailglass/settings/secrets/actions per maintainer note 2026-04-25 (memory `project_hex_api_key.md`). Verify it's environment-scoped (not repo-scoped) per D-27.
