### Unplanned: workflow plumbing fixes
The first push to `origin/main` after Task 1+2 exposed two pre-existing latent bugs that would have prevented Task 3 from working at all:

#### `release-please.yml` — invalid action SHA pin
The pin `googleapis/release-please-action@e63864197e97d1b32d169623835e0544f51e1273` did not exist on the upstream repo. The first run on `3a19b43` failed at "Set up job" with `Unable to resolve action ... unable to find version`. **Fix:** replaced with the verified v4.4.1 commit `5c625bfb5d1ff62eadeeb3772007f7f66fdcf071` per `.planning/research/STACK.md`.

#### `publish-hex.yml` — `publish-admin` ran on a failed `release-please` workflow_run trigger
The job had `if: always() && (publish-core.result == 'success' || publish-core.result == 'skipped') && github.event.inputs.package != 'mailglass'`. On a `workflow_run` trigger, `inputs.package` is empty (`'' != 'mailglass'` is true), and with `publish-core` chained-skip, the guard evaluated true and `publish-admin` ran end-to-end with `HEX_API_KEY` exposed. The actual `mix hex.publish --yes` step was only blocked from publishing because `mailglass` wasn't yet on Hex and Mix dep resolution failed — **pure dumb luck**.

**Fix:** added the same `workflow_dispatch || (workflow_run && conclusion == 'success' && head_branch startsWith 'mailglass-sibling-group-v')` guard that `prepublish-summary` already had to **both** `publish-core` and `publish-admin`. Verified gating works:
- On `70d0306` push: all 4 publish-hex jobs skipped ✓
- On `e26b691` Release-As push: all 4 publish-hex jobs skipped ✓

#### Unplanned: GitHub repo permission
Repo setting `Settings → Actions → General → Workflow permissions → Allow GitHub Actions to create and approve pull requests` was **OFF**, so even after the SHA fix release-please failed at the final step with `GitHub Actions is not permitted to create or approve pull requests`. Enabled via `gh api -X PUT repos/szTheory/mailglass/actions/permissions/workflow -F default_workflow_permissions=write -F can_approve_pull_request_reviews=true`.

## Deviations from Plan

### Mid-plan workflow plumbing fixes (commit `70d0306`)
- **Plan called for:** Tasks 1+2 edits → Task 3 push.
- **Reality:** Tasks 1+2 push exposed two bugs (release-please SHA, publish-hex gating) plus a repo permission setting — all required to be fixed before Task 3 could safely push.
- **Effect:** Plan 10 ended up landing one extra fix commit (`70d0306` — workflow gating + SHA correction) and a one-time API call to enable PR creation. None of these change the plan's stated success criteria; they make the success criteria achievable in practice.
- **Why these weren't caught earlier:** The phase 07-05 / 07.1-08 plans wrote these workflows in-tree and never observed them firing on origin (local-only commits). Per CLAUDE.md "Hex publish only from protected ref + GitHub Environment with required reviewers" expectation, the `hex-publish` environment was auto-created by GitHub on first reference but is not enforcing reviewers. **Defense in depth note:** the user (solo maintainer) chose not to add required reviewers since the gating fix above is the actual safety control and self-approval would just be ceremonial.

### Manifest key normalization (cosmetic, harmless)
Plan 10 Task 1 instructed setting the manifest to `{"mailglass": "0.0.0", "mailglass_admin": "0.0.0"}`. release-please proposes a key normalization in PR #8 to `{".": "0.1.0", "mailglass_admin": "0.1.0"}` because the release-please-config.json packages map uses `.` as the path key for the root package (the `package-name: mailglass` field is metadata, not the manifest key). **No action required** — the PR landing in Plan 11 will fix the format, and PR #8 already proposes both packages at v0.1.0 correctly.
