## Spacing Scale

Declared values (multiples of 4, mapping to Tailwind v4 utilities):

| Token | Value | Tailwind Class | Usage |
|-------|-------|----------------|-------|
| xs | 4px | `p-1` / `gap-1` | Icon inner padding, badge inline gaps |
| sm | 8px | `p-2` / `gap-2` | Compact element spacing, tab bar item padding, form label-to-input gap |
| md | 16px | `p-4` / `gap-4` | Default element spacing, card padding, form field stacking gap |
| lg | 24px | `p-6` / `gap-6` | Sidebar section padding, pane header padding |
| xl | 32px | `p-8` / `gap-8` | Layout-level padding (main pane inset) |
| 2xl | 48px | `p-12` / `gap-12` | Major section breaks |
| 3xl | 64px | `p-16` / `gap-16` | Page-level vertical rhythm (reserved; may not appear in v0.1) |

Exceptions:

- **Touch targets**: all interactive sidebar entries, tab buttons, and toggle buttons must
  be a minimum of 44px tall on mobile (WCAG 2.5.5 Level AAA; brand book ┬¦12). Use
  `min-h-11` (44px) on tap targets even when visual content is smaller.
- **Device toggle buttons** (375 / 768 / 1024): fixed-width 48px ├Ś 32px buttons in a
  segmented control group to keep the three labels scannable side-by-side.
- **Iframe container**: width driven by `@device_width` assign (375 / 768 / 1024px) ŌĆö
  not a spacing token, a dynamic data-driven CSS inline style.

Source: [DEFAULT 8-point grid; brand book ┬¦7.5 for touch target guidance; CONTEXT D-16 for device widths]

## Typography

All sizes in px (rendered via Tailwind v4 `text-*` utilities):

| Role | Family | Size | Weight | Line Height | Usage |
|------|--------|------|--------|-------------|-------|
| Display | Inter Tight | 20px (`text-xl`) | 700 | 1.2 | Mailable name in main pane header, sidebar "Mailers" heading |
| Heading | Inter Tight | 16px (`text-base`) | 600 | 1.3 | Tab section headers, assigns form group headings |
| Body | Inter | 14px (`text-sm`) | 400 | 1.5 | Sidebar scenario names, form labels, metadata, general UI text |
| Label / small | Inter | 12px (`text-xs`) | 500 | 1.4 | Badges, status indicators, mailable module namespace labels |
| Code / mono | IBM Plex Mono | 13px (`text-[13px]`) | 400 | 1.6 | Text tab `<pre>`, Raw tab envelope, Headers table key/value cells, error stacktraces |
| Code / mono bold | IBM Plex Mono | 13px (`text-[13px]`) | 600 | 1.6 | Highlighted header names in Headers tab |

Typography rules:
- **Sentence case** throughout ŌĆö never ALLCAPS for headings or buttons.
- **Tracking**: no extra letter-spacing on body or headings. H1/H2 use `tracking-tight` (`letter-spacing: -0.02em`) per brand book ┬¦7.4 ("slightly condensed in headlines").
- **Glass (#277B96) must never appear as body text on Paper (#F8FBFD)** ŌĆö contrast ratio 4.8:1 passes AA for large text (18px+ or bold 14px+) but fails for small body text. All body text uses Ink or Slate. [RESEARCH ┬¦Accessibility]
- **Max line length** on any prose block (error messages, empty states): `max-w-prose` (Ōēł65ch).

Source: [BRAND ┬¦7.4 typography hierarchy; RESEARCH ┬¦Brand Implementation ┬¦Typography]

### Dark theme

The dark chrome toggle (D-16) switches the surrounding admin UI via `data-theme="mailglass-dark"` on the wrapper element. The preview iframe is **not** inside the theme-scoped element ŌĆö email content renders with its own styles unaffected. Dark theme is admin chrome only at v0.1.

### Visual DON'Ts (enforced in all components)

- No `backdrop-filter: blur()` ŌĆö no glassmorphism
- No `box-shadow: inset` ŌĆö no bevels
- No gradients on functional elements (buttons, tab bars, sidebar items)
- No lens flares, literal broken glass visuals, or frosted overlays
- `--depth: 0` and `--noise: 0` on both daisyUI themes
- Never Glass as a large background area

## Interaction Contract

### State machine: no mailable selected ŌåÆ mailable selected ŌåÆ scenario selected

1. **Index (`/dev/mail/`)**: sidebar visible, main pane shows placeholder copy ("Select a scenario from the sidebar to preview it."). No assigns form. No tabs.
2. **Mailable selected with scenarios (`/dev/mail/:mailable`)**: redirect to first scenario automatically.
3. **Scenario selected (`/dev/mail/:mailable/:scenario`)**: assigns form populated with `preview_props/0` defaults; render fires automatically; HTML tab shown by default; tabs active; device + dark toggles active.
4. **Assigns form change**: `phx-change="assigns_changed"` ŌåÆ server re-renders ŌåÆ iframe/pre refreshed. No debounce at v0.1 (typing triggers immediate re-render; LiveView batches naturally).
5. **Device toggle**: `phx-click="set_device"` ŌåÆ `@device_width` updated ŌåÆ iframe `style` width updated via LiveView push ŌåÆ no full re-render of email content.
6. **Dark toggle**: `phx-click="toggle_dark"` ŌåÆ `@dark_chrome` flipped ŌåÆ `data-theme` attribute on wrapper updated ŌåÆ no re-render.
7. **Tab switch**: `phx-click="set_tab"` ŌåÆ `@active_tab` updated ŌåÆ content area re-renders with the correct artifact; no new Renderer call (all four tab contents pre-computed on render and cached in assigns).
8. **LiveReload broadcast**: re-discover all mailables, re-render current scenario if selected, show flash "Reloaded: {file}".
9. **Reset button**: `phx-click="reset_assigns"` ŌåÆ restore `@current_assigns` from `preview_props/0` defaults ŌåÆ re-render.

### Loading state

No explicit loading spinner at v0.1. Renderer call is synchronous in `handle_event`; LiveView's natural socket latency provides implicit feedback. If render takes >200ms (abnormal), the page simply updates slightly late. Document in MAINTAINING.md: if a mailable takes >200ms to render, profile the template ŌĆö it's the adopter's problem.

### Error recovery

- `preview_props/0` raises on selection: show error card in main pane, sidebar shows `ŌÜĀ` badge. Rest of dashboard remains fully interactive.
- `Mailglass.Renderer.render/1` raises on assigns form change: catch in `handle_event`, show error card in place of the preview. Assigns form remains editable so the adopter can fix the input.
- Mailable module unloads (hot code reload edge case): next LiveReload broadcast triggers re-discovery; if module disappears, remove from sidebar without crash.

### Accessibility interactions

- All interactive elements reachable via keyboard (`Tab` key, `Enter`/`Space` to activate).
- `<details>/<summary>` for sidebar groups: native keyboard expand/collapse.
- Device toggle uses `role="group"` with `aria-pressed` on each button.
- Tab bar uses `role="tablist"` / `role="tab"` / `aria-selected`.
- Focus ring: `outline: 2px solid var(--color-primary); outline-offset: 2px` on `:focus-visible` for all interactive elements.

## Accessibility

Minimum bar: WCAG 2.1 Level AA throughout. Brand book ┬¦12 sets this as the floor.

Verified contrast ratios (all pass AA):

| Pair | Contrast | Passes |
|------|----------|--------|
| Ink `#0D1B2A` on Paper `#F8FBFD` | 15.9:1 | AA + AAA |
| Slate `#5C6B7A` on Paper `#F8FBFD` | 5.1:1 | AA |
| Paper `#F8FBFD` on Glass `#277B96` (button text) | 4.8:1 | AA (large text / UI component) |
| Mist `#EAF6FB` on Ink `#0D1B2A` (sidebar) | 15.1:1 | AA + AAA |
| Signal Amber `#C08A2B` on Paper `#F8FBFD` | 3.6:1 | AA for UI components (not body text) |
| Error Crimson `#B04A3F` on Paper `#F8FBFD` | 5.8:1 | AA |

> Note: Glass on Paper (4.8:1) passes AA for UI components (3:1 threshold) and for large/bold
> text (3:1 threshold) but FAILS for small body text (4.5:1 threshold). **Glass must never
> be used for body-size text on Paper or Mist backgrounds.** Only for button labels
> (bold, 14px+), link text on Paper, or large display text.
