| Boundary | Description |
|----------|-------------|
| Adopter session cookie ↔ mailglass_admin LiveView assigns | The `__session__/2` callback is the SINGLE seam where adopter session data could enter our LiveView. Violated once, every downstream LiveView assign inherits the tainted data. |
| Adopter router pipeline ↔ library routes | The macro emits routes into the adopter's `:browser` pipeline (or whatever the adopter pipes through). No auth/authz enforcement in the library — adopter owns the `:dev_routes` gate. |

|-----------|----------|-----------|-------------|-----------------|

The returned map constructs exactly two keys (`mailables` + `live_session_name`) from opts — never from conn. Test `router_test.exs @tag :session_isolation` asserts `refute Map.has_key?(session, "current_user_id")` + `refute Map.has_key?(session, "csrf_token")` with a seeded test session. |
| T-05-02 | Elevation of Privilege | Adopter mounts mailglass_admin_routes outside `:dev` by mistake | mitigate (documentation-based) | Per CONTEXT D-06, dev-only enforcement is adopter-owned. The macro moduledoc shows the canonical `if Application.compile_env(:my_app, :dev_routes) do ... end` wrapper verbatim. The library explicitly does NOT add `Mix.env()` checks (also verified via grep in acceptance criteria). v0.5 introduces a prod-safe path as a separate, deliberately opt-in macro. |
