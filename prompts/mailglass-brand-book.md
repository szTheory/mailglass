Mailglass Brand Book

Working draft for an open-source Elixir email framework family

Core idea: Mailglass makes email visible.
Not louder. Not shinier. Clearer.

0. Reality check

Before you lock the name in: there is already a public email-related product called Mailglass Lite from HTD Solutions. That does not automatically block an open-source library, but it does mean Mailglass is not a clean-room unique name. Treat the system below as creative direction, not trademark clearance.  ￼

1. Brand in one breath

Mailglass is a Phoenix-native email framework brand for people who want email to feel like real application infrastructure: inspectable, composable, previewable, testable, and calm.

It should feel different from the crowded email-tool landscape built around literal transport names and service-y metaphors. In the same ecosystem you already have names like Courier, Postal, and Postbox; Mailglass stands apart by sounding more like a precision instrument than a delivery company.  ￼

The short version

* Category: developer framework / open-source infrastructure
* Audience: Phoenix and Elixir teams
* Promise: make email understandable end to end
* Mood: clear, technical, humane, composed
* Not: salesy, growth-hacky, ESP-ish, “AI magic”

2. Brand essence

Brand idea

Mailglass = mail you can see through.

The name works because it joins two useful ideas:

* Mail = delivery, messages, correspondence, routing, headers, inboxes, feedback loops
* Glass = clarity, preview, panes, observability, transparency, inspection, trust

That second half is the differentiator.
Mailglass should never feel like “send more campaigns.”
It should feel like understanding a complicated system through a clean pane.

Brand promise

Mailglass turns email from a black box into a visible system.

Positioning line

The clear email framework for Phoenix.

Alternate taglines

* Email, made visible.
* Compose. Preview. Deliver. Observe.
* Clearer email for Phoenix apps.
* From template to telemetry.
* The missing email layer for Phoenix.

3. What the brand stands for

3.1 Clarity

Mailglass explains what is happening.
It favors visible state, previews, inspectable flows, legible UI, legible docs, and legible errors.

3.2 Composure

It is serious without being stiff.
The brand should feel calm under pressure: no neon chaos, no hacker theatrics, no fake urgency.

3.3 Precision

Mailglass is builder-facing.
It should feel exact, typed, predictable, and trustworthy.

3.4 Warmth

It is infrastructure, but it is not cold.
There should be a little craft in the typography, a little softness in the language, and a little generosity in the docs.

3.5 Visibility over magic

Mailglass should not present itself as wizardry.
It should present itself as a well-lit workbench.

4. Brand personality

Think of Mailglass as:

* clear, not cloudy
* exact, not fussy
* confident, not cocky
* warm, not cute
* modern, not trendy
* technical, not intimidating

Personality sliders

* Formal ↔ Casual: slightly formal
* Human ↔ Machine: balanced, with a human edge
* Quiet ↔ Loud: quiet
* Playful ↔ Serious: serious with a light hand
* Minimal ↔ Decorative: minimal
* Utility ↔ Luxury: utility with taste

5. Voice and tone

Voice

Mailglass speaks like a thoughtful maintainer.

It uses plain English, exact nouns, strong verbs, and short sentences. It does not inflate. It does not posture. It does not market-speak like a commercial ESP. It sounds like someone who understands email deeply and respects the reader’s time.

Tone principles

1. Clear first

Prefer the direct word over the clever one.

* Say preview
* Not experience the full rendering lifecycle

2. Explain, don’t hype

Prefer confident understatement.

* Say provider-normalized events
* Not revolutionary observability

3. Calm under failure

Errors should be specific and composed.

* Say Delivery blocked: recipient is on the suppression list
* Not Oops! Something went wrong

4. Generous with context

Docs and UI copy should help the reader recover quickly.

* Say This message was skipped because the recipient previously unsubscribed
* Not Suppressed

What Mailglass sounds like

* “Render a real message before you send it.”
* “Inspect the headers, HTML, text part, and delivery events in one place.”
* “Use the preview with realistic props.”
* “This route matched the inbound recipient pattern.”

What Mailglass never sounds like

* “10x your outreach”
* “supercharge growth”
* “AI-powered everything”
* “crush your email KPIs”
* “beautifully effortless magic”

One-line description

README blurb

Mailglass brings structure and visibility to email in Elixir apps: typed mailables, component-based templates, previews, delivery workflows, and a foundation for feedback, routing, and administration.

Elevator pitch

Phoenix has a mailer primitive.
Mailglass aims to be the missing framework around it: the layer that makes email easier to build, easier to preview, easier to test, and easier to trust in production.

7. Visual identity

7.1 Visual concept

The visual metaphor is clarity through panes.

Not literal glass everywhere.
Not heavy glassmorphism.
Not glossy reflections.

Use glass as a metaphor for:

* preview surfaces
* layered visibility
* clean boundaries
* light passing through structure
* seeing systems without noise

Rule: use translucency sparingly.
Mailglass should feel crisp first, glassy second.

7.2 Logo direction

Recommendation

Lead with a wordmark-first identity.
Mailglass is a framework brand, so the name should do the heavy lifting.

Wordmark

Preferred expression:

* Mailglass in prose
* mailglass as a logo lockup option

Why lowercase works:

* softer
* more contemporary
* less corporate
* visually smoother with a strong sans serif

Why title case still matters:

* cleaner in docs, package names, release notes, and references

Symbol direction

Use a symbol only as a secondary mark.

Best icon concept

A pane/window form with a subtle envelope fold or message flap implied through negative space.

The symbol should suggest:

* visibility
* containment
* routing
* inspection
* message structure

It should not look like:

* a paper airplane
* a flying envelope
* a mailbox on a post
* a chat bubble
* a generic send arrow
* a “glossy app icon” from 2014

Logo behavior

* Prefer flat color
* Prefer one or two colors only
* No bevels
* No heavy highlights
* No fake reflections
* No 3D chrome
* No gradients inside tiny marks unless used very subtly at large sizes

Clear space

Give the mark room to breathe.
Minimum clear space should feel generous rather than mathematically cramped.

Minimum size

* icon only: never smaller than favicons/app icons require
* wordmark: never so small that the gla cluster becomes muddy

7.3 Color system

The Mailglass palette should feel like ink, glass, frost, and signal.

Core palette

Ink — #0D1B2A
Primary text, dark UI chrome, footer backgrounds, code-adjacent surfaces.

Glass — #277B96
Primary brand accent. Links, active states, accents, highlights, selected items.

Ice — #A6EAF2
Bright supporting accent for dark surfaces, soft highlights, illustrations.

Mist — #EAF6FB
Secondary background, subtle cards, preview surfaces, frosted layers.

Paper — #F8FBFD
Primary light background.

Slate — #5C6B7A
Secondary text, metadata, subdued UI.

Supporting semantic colors

Signal Amber — #A95F10
Warnings, caution states, operational notices.

Error Crimson — #B42318
Errors, destructive states, critical warnings.

Success Pine — #166534
Healthy delivery states, successful config checks, completed actions.

Color rules

* Ink + Paper is the default reading pair.
* Glass is the brand accent, not the background for everything.
* Mist is for quiet depth, not washed-out vagueness.
* Ice belongs on dark surfaces more than on white.
* Avoid rainbow dashboards. Mailglass should feel disciplined.

The accessibility baseline should follow WCAG guidance: normal text should meet at least 4.5:1 contrast, large text at least 3:1, and meaningful UI graphics/components at least 3:1 against adjacent colors.  ￼

Recommended usage split

* 55% Paper / light neutrals
* 25% Ink / dark structure
* 12% Glass / primary brand accent
* 5% Mist / layered surfaces
* 3% semantic colors

7.4 Typography

Use typography that feels screen-native, highly legible, and open-source aligned.

Inter is an excellent primary family because it is explicitly designed for computer screens and uses a tall x-height for readability. Inter Tight works well for display lines. IBM Plex Mono is a strong monospace companion because IBM describes Plex as an open-source family designed to work well in UI environments; it brings just enough technical character without going full terminal cosplay. These families also fit the ethos of an open-source project because they are open-source and straightforward to use and self-host.  ￼

Recommended stack

* Display: Inter Tight
* UI + body: Inter
* Code / CLI / tokens / labels: IBM Plex Mono

Typographic feel

* clean
* slightly condensed in headlines
* roomy in body copy
* technical where it matters
* never sterile

Suggested hierarchy

* H1: Inter Tight, 700, 40–48px
* H2: Inter Tight, 700, 30–36px
* H3: Inter, 600, 22–26px
* Body: Inter, 16–18px
* Small: Inter, 14px
* Code / token text: IBM Plex Mono, 13–14px

Typographic rules

* Keep line length moderate
* Prefer sentence case over all caps
* Use mono for code, RFCs, env vars, adapters, statuses, and structured values
* Avoid overly tight tracking
* Avoid giant oversized hero type that makes the project feel consumer-SaaS-y

7.5 Iconography

Mailglass icons should feel like product instrumentation.

Style

* simple geometry
* mostly outline icons
* slightly rounded corners
* consistent stroke weight
* minimal fills
* no cartoon metaphors

Motifs that fit

* pane / window
* flap / fold
* route / path
* stack / layers
* eye / inspect
* headers / lines / fields
* timeline / event markers
* shield / trust
* toggle / switch / state

Motifs to avoid

* paper planes
* megaphones
* rockets
* sparkles
* shouting notification bells
* mascot faces
* generic “marketing” arrows

7.6 Illustration

Illustration, if used at all, should be diagrammatic rather than whimsical.

Think:

* message flow
* layered panes
* header fields
* route branching
* preview frames
* status timelines
* compliance checks
* event normalization

Illustration should feel like a friendly systems diagram, not a startup landing page cartoon.

7.7 Imagery

Acceptable imagery

* close, quiet product screenshots
* cropped UI panes
* layered transparent panels
* abstract route maps
* message headers as texture
* architectural glass, reflections, grids, shadows
* desks, keyboards, notebooks, terminals, but used sparingly
* detail shots rather than lifestyle stock

Avoid

* smiling support teams in headsets
* people pointing at dashboards
* paper planes
* floating envelopes
* loud 3D blobs
* crypto-looking neon gradients
* fake holograms
* “growth” stock imagery
* anything that makes the project feel like outbound-sales software

Photo treatment

If photography is used:

* keep it cool-toned or neutral
* low drama
* crisp contrast
* minimal clutter
* preferably structural rather than sentimental

Accessibility note for imagery

Decorative images should be treated as decorative. Functional images should be described by the action or purpose they trigger, not by their visual appearance, and empty alt="" is appropriate for purely decorative images.  ￼

7.8 Motion

Mailglass motion should be measured and useful.

Motion style

* short fades
* subtle slides
* pane reveals
* state transitions
* timeline progressions

Avoid

* bounce
* elastic overshoot
* confetti
* dramatic parallax
* flashy onboarding flourishes

Motion feeling

A pane opened.
A state became visible.
A system updated.
That’s the vibe.

8. Brand behavior in product surfaces

8.1 Docs site

Docs should feel like the purest expression of the brand:

* white or paper base
* strong ink typography
* restrained glass accent
* mono used for structure and code
* screenshots with minimal chrome
* examples before slogans

Docs headline style

Prefer:

* “Preview a message locally”
* “Normalize provider events”
* “Route inbound mail with a mailbox module”

Avoid:

* “Unlock your email future”
* “Ship world-class communication experiences”

8.2 GitHub

Mailglass on GitHub should feel disciplined and maintainer-friendly.

Repo visuals

* icon-on-ink avatar
* simple social preview image
* one concise promise line
* no decorative clutter

README structure

1. what it is
2. why it exists
3. example
4. package family
5. install
6. preview / screenshot
7. roadmap
8. philosophy

8.3 Hex package pages

Use the same naming pattern across the family:

* mailglass
* mailglass_preview
* mailglass_inbound
* mailglass_campaigns
* mailglass_deliverability

Keep visual consistency across package docs so the family reads as one system.

8.4 Dashboard UI

If you build preview/admin UI later, the product surface should become the living brand book.

The UI should feel like:

* a lab bench
* a mail console
* a composed admin panel
* not an analytics casino

8.5 CLI and terminal copy

CLI output should be plain, useful, and minimally branded.

Good:

* preview started at /dev/mail
* delivered via postmark
* suppressed: recipient previously unsubscribed

Bad:

* ✨ your email journey begins now ✨

9. Naming architecture for the family

Keep the family literal and modular.

Master brand

Mailglass

Product / package extensions

* Mailglass Preview
* Mailglass Inbound
* Mailglass Campaigns
* Mailglass Deliverability
* Mailglass Admin
* Mailglass Components
* Mailglass Test
* Mailglass Webhooks

Naming rule

The parent brand is the poetic layer.
The child packages are the practical layer.

That means:

* parent = evocative
* package names = obvious

Do not create cute sub-brand names unless there is a very strong reason.
This ecosystem will benefit more from coherence than from cleverness.

11. Logo and visual don’ts

Never do these:

* heavy glassmorphism
* glossy chrome effects
* bevels
* lens flares
* literal broken glass
* mailbird / paper plane lookalikes
* too much cyan-on-cyan
* a generic blue SaaS palette with no ink depth
* making everything transparent because the name has “glass” in it

Use glass as metaphor, not gimmick.

12. Accessibility guardrails

Mailglass should visibly care about accessibility. That is on-brand.

WCAG’s baseline guidance is straightforward: normal text needs at least 4.5:1 contrast, large text 3:1, and meaningful UI graphics/components 3:1. Use those as minimums, not aspirations. Alternative text should reflect whether an image is informative, functional, or decorative.  ￼

Practical rules

* never rely on color alone for state
* links should look like links
* focus states should be obvious
* screenshots should be legible
* decorative imagery should stay decorative
* UI chrome should never outrank content

If everything else gets lost, preserve these five rules:

2. Visibility beats magic.
3. Calm beats hype.
4. Precision beats ornament.
